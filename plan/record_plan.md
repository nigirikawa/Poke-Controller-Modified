# Poke-Controller 録画機能 拡張設計仕様書

## 1. アーキテクチャ概要

既存のカメラフレーム配信フローを拡張し、録画専用のサブスレッド（Producer-Consumerパターン）を導入します。メインのコマンド実行ループからメソッドを呼び出すことで、非同期での録画処理と、同期的なファイル管理を実現します。

* **Producer (CameraThread)**: 既存のフレーム取得処理。録画実行中フラグがONの場合、録画用キューにもフレームをノンブロッキング（`put_nowait` 等）で供給します。
* **Consumer (VideoRecorderThread)**: 録画用キューからフレームを取り出し、ディスクへのI/O処理（動画書き出し）を専任で行います。
* **Controller (Main Loop)**: `BaseExeTrade` 等のマクロのメインループにて、録画の開始・終了、および古いファイルの削除処理を制御します。

---

## 2. モジュール詳細仕様

### 2.1 録画機能 (VideoRecorder)

サブスレッドで動作し、OpenCVの `VideoWriter` を用いてフレームのエンコードとファイル書き出しを行います。

* **動画フォーマットとコーデック**:
    * **コンテナ**: `.avi`
    * **コーデック**: `MJPG` (`cv2.VideoWriter_fourcc(*'MJPG')`)
    * **選定理由**: 処理負荷が低く、リアルタイムでのエンコードに適しているため、メインスレッドへの影響を最小限に抑えられます。
* **画質・解像度**:
    * 解像度: `1280 x 720` (既存の `self.camera.capture` と同等)
    * FPS: メニューで選択された現在のFPS値を動的に取得して設定。
* **インターフェース**:
    * `start_recording() -> None`:
        * 録画用サブスレッドを起動（またはアクティブ化）し、録画用キューへのフレーム受け入れを開始します。
        * 内部で一意なファイル名（タイムスタンプ等）を生成し、`VideoWriter` を初期化します。
    * `stop_recording() -> str`:
        * カメラからのフレーム供給フラグをOFFにします。
        * キュー内に残っているフレームの書き出し完了を待ちます（サブスレッドの終了同期）。
        * `VideoWriter.release()` を実行し、動画ファイルをクローズします。
        * **戻り値**: 書き出しが完了した動画ファイルの絶対パス。

### 2.2 録画ファイル管理機能 (ClipManager)

メインループ内で同期的に呼び出される、録画済みファイルの世代管理コンポーネントです。

* **データ構造**:
    * `collections.deque(maxlen=10)` などをラップし、ファイルパスのリストを保持します。
* **削除ロジック**:
    * `stop_recording()` から受け取った新しい動画パスをキューに登録します。
    * 登録直前にキューの要素数が10個に達している場合、最も古いファイルパスを取り出し、ファイルシステム上から物理削除（`os.remove` または `pathlib.Path.unlink`）します。
    * I/O完了を待機し、**削除処理が完全に終わってからメインループの次の処理へ進む**よう同期制御を行います。

## 3. 制御フロー設計

### 3.1 正常系のループ制御

マクロループ（`while True:`）の境界において、以下のように制御をフックします。これにより「1ループ＝1動画クリップ」として管理されます。

```python
def do(self):
    while True:
        # 【1】前ループのクリーンアップ確認・新しい録画の開始
        self.recorder.start_recording()
        
        try:
            # メイン処理（既存の send_trade / recv_trade 等）
            self.send_trade()
            
        except InitializationError:
            # エラー時の回復処理
            self.reset_to_main_menu(10)
            
        finally:
            # 【2】ループ終了時（正常・エラー復帰問わず）に録画停止
            # ※次のループに入る前に必ず完了させる
            video_path = self.recorder.stop_recording()
            
            # 【3】クリップ管理と古いファイルの同期削除
            if video_path:
                self.clip_manager.add_and_rotate(video_path)
```

### 3.2 異常系（例外発生時のプログラム停止）の保護

予期せぬ `Exception`（画像認識の致命的な失敗、ライブラリのクラッシュなど）でプログラムがダウンする際、書き込み中の動画ファイルが破損（ヘッダ未完等）するのを防ぐためのフェイルセーフを実装します。

```python
def do(self):
    try:
        while True:
            self.recorder.start_recording()
            try:
                self.send_trade()
            except InitializationError:
                self.reset_to_main_menu(10)
            finally:
                video_path = self.recorder.stop_recording()
                if video_path:
                    self.clip_manager.add_and_rotate(video_path)
                    
    except Exception as e:
        # 【重要】致命的な例外でループを抜けた場合の最終クリーンアップ
        # 録画が実行中のまま落ちた場合は強制終了・保存を行う
        if self.recorder.is_recording():
            saved_path = self.recorder.stop_recording()
            # 必要に応じてエラー時の動画は別フォルダに退避、またはログにパスを出力
            logger.error(f"プログラム異常終了。エラー直前の録画を保存しました: {saved_path}")
        raise e  # 元の例外を再送出してプログラムをダウンさせる
```

## 4. 検討・考慮事項

1.  **フレームドロップの許容**:
    * エンコード（書き込み）処理が現在のFPSに追いつかず録画用キューが満杯になった場合、メインスレッド側の負荷を上げないために**「古いフレームをスキップ（ドロップ）する」**方針を推奨します。動画のカクつきは発生しますが、マクロ自体の動作安定性を優先するためです。
2.  **I/Oブロッキング**:
    * ファイルの物理削除（`unlink`）はメインスレッドで同期的に行いますが、通常はミリ秒単位で終わるため、マクロの待機時間（sleep等）の間に吸収可能と想定しています。
